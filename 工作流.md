# 一、基本流程

## 1. Plan 阶段：（如必要）

```text
进入 Plan 模式。本阶段只分析，不修改文件。

任务：【这里写任务】

请先阅读：
- AGENTS.md
- docs/PRD.md
- docs/architecture.md
- docs/markdown_spec.md
- docs/progress.md（如果存在）
- 跨项目开发准则 v1.1.md（如果存在）

只输出以下内容：

## 结论
一句话说明建议怎么做。

## 涉及文件
| 文件 | 作用 | 是否需要修改 |
|---|---|---|

## 推荐方案
最多 5 条。

## 风险点
最多 5 条。

## 验收清单
最多 8 条。

## 需要确认的问题
没有就写“无”。

## 给 ChatGPT 的摘要
控制在 20 行以内，只保留关键信息。

输出限制：
1. 不要输出完整代码。
2. 不要输出完整 diff。
3. 不要解释太多背景。
4. 总输出控制在 80 行以内。
```

------

## 2. Build 阶段：

```text
进入 Build 模式。按已确认方案执行。

任务：【这里写任务】

限制：
1. 只修改 Plan 阶段确认的文件。
2. 不做无关重构。
3. 不新增未确认依赖。
4. 不破坏 Markdown 保存闭环。
5. 不破坏图片资产链路。
6. 不破坏写作模式生命周期。
7. 不破坏窗口关闭守卫。
8. 完成后运行 pnpm build。

输出格式：

## 结论
完成 / 未完成 / 部分完成。

## 改动文件
| 文件 | 改动 |
|---|---|

## 验证命令
| 命令 | 结果 |
|---|---|

## 风险/未完成
最多 5 条。

## 给 ChatGPT 的摘要
控制在 20 行以内。

输出限制：
1. 不输出完整代码。
2. 不输出完整 diff。
3. 不输出无关解释。
4. 如果 pnpm build 报错，只贴最后 80 行错误。进入 Build 模式。按已确认方案执行。

限制：
1. 只修改已确认文件。
2. 不做无关重构。
3. 不新增未确认依赖。
4. 不破坏图片资产链路。
5. 不破坏 Markdown 保存闭环。
6. 不破坏窗口关闭守卫。
7. 完成后运行 pnpm build。

输出：
1. 改动文件。
2. 核心逻辑。
3. 已运行命令。
4. 手动验证点。
5. 风险/未完成。
```

------

## 3. Review 阶段：（如必要）

```text
进入 Review 模式。不要修改代码。

请审查刚才的改动。

只检查：
1. git diff --stat
2. git diff --name-only
3. 是否修改无关文件
4. 是否影响 Markdown 正文
5. 是否影响图片路径
6. 是否影响写作模式
7. 是否影响保存/自动保存
8. 是否影响导出
9. 是否影响窗口关闭守卫
10. 是否建议提交

输出格式：

## 是否建议提交
是 / 否 / 需要修复后提交。

## diff 概览
贴 git diff --stat 结果。

## 风险检查
| 检查项 | 结果 | 说明 |
|---|---|---|

## 必须手动验证
最多 8 条。

## 给 ChatGPT 的摘要
控制在 20 行以内。

输出限制：
1. 不输出完整 diff。
2. 不输出完整代码。
3. 总输出控制在 80 行以内。
```

# 二、提交策略也要固定

每个功能必须有 checkpoint：

```powershell
pnpm build
git status
git diff --stat
git add .
git commit -m "type: message"
```

建议 commit 类型：

| 类型        | 用法             |
| ----------- | ---------------- |
| `feat:`     | 新功能           |
| `fix:`      | bug 修复         |
| `refactor:` | 重构             |
| `style:`    | UI/CSS           |
| `chore:`    | 配置、checkpoint |
| `test:`     | 测试样本         |

图片链路、关闭守卫这类稳定点必须单独提交，不要和新功能混在一个 commit。

------

# 三、工作组合

```text
你：产品判断 + 最终验收
我：架构边界 + 风险控制 + 指令设计
OpenCode Plan：读取真实代码，制定方案
OpenCode Build：小步实现
OpenCode Review：审查 diff 和回归风险
Git：每步 checkpoint
```

这样既能利用 OpenCode 的代码上下文能力，又不会让它在复杂链路里自由发挥。
